**Goal:** Ship `python-max-bot` v0.3 — webhook subscription endpoints + bot introspection + action indicators — per `docs/design/0003-v0.3.md`.

```python
"""Webhook subscription model — response from GET /subscriptions."""

from __future__ import annotations

from pydantic import BaseModel, ConfigDict


class Subscription(BaseModel):
    """An active webhook subscription.

    `update_types` uses `list[str]` rather than `list[UpdateType]` so a
    future server-side addition never crashes a get_subscriptions call.
    Request-side filtering (subscribe(update_types=...)) uses the strict
    UpdateType enum because the user controls that input.
    """

    model_config = ConfigDict(extra="ignore")

    url: str
    time: int
    update_types: list[str] | None = None
```

```python
class MaxBadResponseError(MaxError):
    """The API returned 2xx with `{success: false}` in the body.

    Synthesized from the response body — does not carry an
    httpx.Response, since the underlying HTTP call succeeded. Raised
    from subscribe / unsubscribe / send_action when the server reports
    a soft failure on an otherwise-successful HTTP response.
    """

    def __init__(self, message: str | None) -> None:
        super().__init__(message or "API reported success=false with no message")
        self.message = message
```

Add the validation helpers in the `# ── Internal helpers ─` section
near the bottom of the class:

```python
    @staticmethod
    def _validate_webhook_url(url: str) -> None:
        if not url.startswith("https://"):
            raise ValueError("webhook url must start with https://")

    @staticmethod
    def _validate_webhook_secret(secret: str) -> None:
        if not 5 <= len(secret) <= 256:
            raise ValueError("webhook secret length must be between 5 and 256 characters")
        # Charset: A-Z, a-z, 0-9, hyphen
        if not all(c.isalnum() or c == "-" for c in secret):
            raise ValueError("webhook secret may only contain A-Z, a-z, 0-9, and hyphen")
```

Note `c.isalnum()` accepts Unicode letters/digits, while the API
charset is strictly ASCII. Since the length check happens first and
typical user mistakes (special chars, spaces, slashes) are caught by
the alphanumeric check, this is a pragmatic match for the documented
intent. If a user passes a Cyrillic letter it'll pass our check and the
API will reject it — surface there, not here.
