# python-max-bot v0.1 — Design

**Status:** approved
**Date:** 2026-05-04
**Author:** Kitaeza
**AI co-author:** Claude (Anthropic). This project is developed end-to-end
in collaboration with Claude — design, code, and tests. Commits are
co-authored; the README and PyPI description state this explicitly so
downstream users know what they are adopting.

## Goal

A thin, stable, async Python client for the Max messenger Bot API
(`https://platform-api.max.ru`). Targets developers who want a predictable
v1.0-track surface — not a fast-moving framework — for sending and editing
messages, uploading attachments, and polling for updates.

## Non-Goals (v0.1)

- No bot framework (handlers, dispatchers, FSM, middleware). Those belong
  to a higher layer; users compose them on top of the client themselves.
- No webhook server (only the API call to register/unregister webhooks
  ships in a later release).
- No automatic retries (intentionally deferred to v0.2 — see Roadmap).
- No sync flavor. Async-only; users running sync code wrap with
  `asyncio.run`.

## Architecture

A single `MaxClient` async class fronts an internal httpx-based transport
layer. Pydantic v2 models describe every request body and response, exported
flat from `max_bot_api`.

The package is small enough (~10 public methods in v0.1) that
resource-grouping (`client.messages.send()`, `client.chats.get()`) would be
over-engineering. All methods sit directly on `MaxClient`.

## Public API

All methods are async; all optional parameters are keyword-only.

### Construction & lifecycle

```python
class MaxClient:
    def __init__(
        self,
        token: str,
        *,
        base_url: str = "https://platform-api.max.ru",
        timeout: float = 30.0,
        transport: httpx.AsyncBaseTransport | None = None,  # for testing
    ) -> None: ...

    async def __aenter__(self) -> "MaxClient": ...
    async def __aexit__(self, *exc) -> None: ...
    async def aclose(self) -> None: ...
```

### Validation rules

- `chat_id` and `user_id` on `send_message` are mutually exclusive. Raising
  `ValueError` *before* any HTTP call keeps the failure local and cheap.
- `text` and `attachments` cannot both be empty on `send_message` /
  `edit_message`. The API rejects this anyway; we surface it earlier.
- `text` longer than 4000 characters raises `ValueError`. The API rejects
  it; surface earlier with a clear message.

### Typical usage

```python
async with MaxClient(token) as client:
    img = await client.upload_image(open("photo.jpg", "rb").read())
    await client.send_message(
        chat_id=42,
        text="<b>Hello Max</b>",
        format="html",
        attachments=[img],
    )
```

## Error handling

A single hierarchy under `MaxError`. Users catch `MaxAPIError` for any
non-2xx response or `MaxError` for everything (including transport).

```python
class MaxError(Exception):
    """Base for everything raised by python-max-bot."""

class MaxAPIError(MaxError):
    """Any non-2xx response from the Max API."""
    status_code: int
    code: str | None              # the API's own error code, if present
    message: str
    response: httpx.Response      # for advanced debugging

class MaxAuthError(MaxAPIError):           # 401
class MaxNotFoundError(MaxAPIError):       # 404
class MaxValidationError(MaxAPIError):     # 400
class MaxMethodNotAllowedError(MaxAPIError):  # 405
class MaxRateLimitError(MaxAPIError):      # 429
    retry_after: float | None     # parsed from Retry-After header
class MaxServerError(MaxAPIError):         # 5xx
class MaxServiceUnavailableError(MaxServerError):  # 503

class MaxTransportError(MaxError):
    """Wraps httpx.TransportError — network failed, no HTTP response."""

class MaxTimeoutError(MaxTransportError):
    """Wraps httpx.TimeoutException."""
```

### Mapping rules

- 2xx → return the parsed body.
- 4xx / 5xx → parse `{"code", "message"}` from the body when present, raise
  the matching subclass.
- httpx network/timeout exceptions → wrap as `MaxTransportError` /
  `MaxTimeoutError` so users only need one import root.
- Pydantic validation failure on a response body → re-raise as
  `MaxAPIError` with the original `ValidationError` chained as
  `__cause__`. A schema drift on the server side is something users need to
  see loudly, not a silent garbage object.

## Testing strategy

All tests in v0.1 are unit tests; no live API hits in CI.

### Test layers

1. **Models** (`tests/test_models.py`)
   - Round-trip a representative payload from the Max docs through every
     model: parse → dump → assert equality. Catches typos in field names
     and missed `serialization_alias` values.
   - Discriminated-union edge cases: every attachment subtype, every
     update event subtype.
   - Required-field tests: raise on missing required, accept missing
     optional.

2. **Client methods** (`tests/test_client.py`) — using **respx** to mock
   httpx
   - For each public method: one happy path, one 4xx path, one 5xx path.
   - Verify the *outgoing* request: URL, query params, headers
     (`Authorization: <token>` exact form — no `Bearer` prefix), body.
   - Mutual-exclusion guards: `chat_id` + `user_id` both set or both
     unset → `ValueError` *before* any HTTP call.
   - `upload_image` performs the two-call dance correctly: request URL →
     POST file → returns an attachment carrying the right token.

3. **Exceptions** (`tests/test_exceptions.py`)
   - Each HTTP status code → assert the right subclass and that
     `status_code` / `code` / `message` / `response` are populated.
   - `Retry-After` → parsed onto `MaxRateLimitError.retry_after`.
   - Network failure (respx-simulated) → `MaxTransportError`.

### CI matrix

- Python 3.10, 3.11, 3.12, 3.13
- Ubuntu only. Pure-Python lib; httpx handles platform differences. Add
  Windows / macOS later if a user reports a real issue.

### Coverage gate

90% line coverage in CI. Easy to hit with the layers above and useful for
flagging untested branches early.

### Out of scope for v0.1

- Live integration tests against a real bot (need a token in CI, a sandbox
  channel, secret handling). Add later as an opt-in workflow.
- Property-based tests. Useful but not needed at this surface size.
- Performance benchmarks. Nothing perf-sensitive.

## Repo, release, and publish

### Branch model

Trunk-based. `main` is always releasable. Feature branches are short-lived,
merge via PR.

### Versioning

SemVer. Version stored in `src/max_bot_api/_version.py` as
`__version__ = "0.1.0"`. Pre-1.0: minor versions may break the API, signaled
clearly in the CHANGELOG.

### Release flow

1. Bump `_version.py`, update `CHANGELOG.md`, commit on `main`.
2. `git tag v0.1.0 && git push --tags`.
3. `release.yml` triggers on `v*`:
   - Lint, typecheck, test (sanity)
   - `uv build` → sdist + wheel into `dist/`
   - `uv publish` to PyPI via **trusted publisher (OIDC)** — no API token
     stored in GitHub secrets, no credential rotation
   - Create a GitHub Release from the tag, body = matching CHANGELOG entry

### One-time setup

- Reserve `python-max-bot` on PyPI by uploading `0.1.0a0` once `pyproject.toml`
  is in place.
- Configure the trusted publisher on PyPI (web UI: GitHub repo +
  `release.yml`).

## Roadmap

### v0.2

- **`RetryPolicy`** — opt-in retry with configurable backoff. The default
  policy: retry on `MaxServerError` and `MaxTransportError`, exponential
  backoff (1s, 2s, 4s), max 3 attempts, never retry on 4xx (except 429,
  which respects `retry_after`). Exposed via the `MaxClient` constructor:
  `MaxClient(token, retry=RetryPolicy(max_attempts=3, ...))`.
  See `docs/design/0002-v0.2-retry-policy.md` for the v0.2 design.
  **Status: shipped in v0.2.0.**
- Live integration test workflow (manual trigger).

### v0.3+

- Webhook subscription endpoints (set/get/delete subscription).
- Bot introspection: `get_me`, `get_chat_members`, `get_chat_admins`.
- Action indicators (typing, sending photo, etc.).

### v1.0

API freeze, breaking changes only on major version bumps. No new feature
gating; whatever surface is mature and stable at the time gets stamped.
