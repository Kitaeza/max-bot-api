# python-max-bot v0.2 — RetryPolicy Design

**Status:** approved
**Date:** 2026-05-04
**Author:** Kitaeza
**AI co-author:** Claude (Anthropic)
**Tracks:** Roadmap §v0.2 in `docs/design/0001-v0.1.md`

## Goal

Ship opt-in, configurable request retries for `MaxClient`. The default
behavior must remain identical to v0.1 — a `MaxClient` constructed
without a `retry` argument issues exactly one HTTP call per public
method, just as today.

## Non-Goals

- No global, mandatory retry. Retry stays opt-in for v0.2.
- No per-call override (e.g., `send_message(..., retry=...)`). The policy
  is set once at client construction. A per-call knob is a v0.3+ topic
  if anyone asks.
- No request-deduplication / idempotency-key support. The Max API does
  not expose idempotency keys; we only protect non-idempotent methods by
  *not* retrying them on 5xx (see Semantics).
- No retry hook / callback for observability. Add non-breakingly later
  if requested.
- No live integration test workflow. The roadmap groups it with v0.2,
  but it ships separately.

## Public API

```python
# src/max_bot_api/retry.py
from dataclasses import dataclass

@dataclass(frozen=True)
class RetryPolicy:
    max_attempts: int = 3            # total attempts including the first
    backoff_initial: float = 1.0     # seconds before retry #2
    backoff_multiplier: float = 2.0  # each subsequent wait *= this
    backoff_max: float = 30.0        # cap on a single wait
    jitter: bool = True              # ±25% randomization, on by default
```

`MaxClient.__init__` gains one keyword-only argument:

```python
def __init__(
    self,
    token: str,
    *,
    base_url: str = "https://platform-api.max.ru",
    timeout: float = 30.0,
    transport: httpx.AsyncBaseTransport | None = None,
    retry: RetryPolicy | None = None,   # NEW; None ⇒ v0.1 behavior
) -> None: ...
```

Re-exported flat from `max_bot_api`: `RetryPolicy`.

`RetryPolicy` is a frozen dataclass — hashable, immutable, no Pydantic
overhead. It is configuration, not a wire model.

## Retry semantics

The policy decides which exceptions are retry-eligible and how long to
wait between attempts.

| Failure | Retry-eligible? | Notes |
|---|---|---|
| `MaxTransportError` (incl. `MaxTimeoutError`) | always | No HTTP response was received → safe by definition |
| `MaxServerError` (5xx, incl. `MaxServiceUnavailableError`) | only if call site passes `idempotent=True` | Protects writes from double-applying when the server processed the request but the response was lost |
| `MaxRateLimitError` (429) | always | Sleep is `max(retry_after_header, computed_backoff)`; `retry_after` is **not** clamped by `backoff_max` — the server's instruction wins |
| Any other 4xx (`MaxAuthError`, `MaxValidationError`, `MaxNotFoundError`, `MaxMethodNotAllowedError`) | never | Retrying won't change the answer |

The last attempt re-raises the underlying exception unmodified — callers
see the same error type as in v0.1.

### Backoff calculation

For attempt `N` (1-indexed), the wait *before* attempting `N+1` is:

```python
base = min(backoff_initial * backoff_multiplier ** (N - 1), backoff_max)
wait = base * uniform(0.75, 1.25) if jitter else base
```

With defaults and no jitter that yields **1s, 2s, 4s** — matching the
sequence called out in the v0.1 roadmap. With `backoff_max=30.0` the
sequence saturates rather than growing without bound.

The Max API's `Retry-After` header is parsed onto
`MaxRateLimitError.retry_after` already (v0.1). When the policy wakes
from a 429 it uses `max(retry_after, backoff(N))` — never less than the
server asked for, never less than the policy's own pacing.

### `policy is None` short-circuit

When `retry` is `None`, the transport bypasses the loop entirely and
issues a single request — byte-for-byte identical to v0.1 behavior. No
sleep call, no jitter import, no retry-state object. This keeps the
default-path overhead at zero and means existing v0.1 users see no
behavior change on upgrade.

## Transport changes

`Transport.request(...)` gains a keyword-only argument:

```python
async def request(
    self,
    method: str,
    path: str,
    *,
    ...,
    idempotent: bool = False,   # NEW; conservative default
) -> dict[str, Any]: ...
```

The default is `False` so any forgotten call site fails safe (no retry
on 5xx) instead of silently double-applying.

The retry loop lives in `Transport.request`. Outline:

```python
if self._retry is None:
    return await self._do_request(...)

policy = self._retry
for attempt in range(1, policy.max_attempts + 1):
    try:
        return await self._do_request(...)
    except MaxRateLimitError as e:
        if attempt == policy.max_attempts:
            raise
        wait = max(e.retry_after or 0.0, _backoff(policy, attempt))
        await asyncio.sleep(wait)
    except MaxServerError:
        if attempt == policy.max_attempts or not idempotent:
            raise
        await asyncio.sleep(_backoff(policy, attempt))
    except MaxTransportError:
        if attempt == policy.max_attempts:
            raise
        await asyncio.sleep(_backoff(policy, attempt))
```

`_backoff(policy, attempt)` is a free function in `retry.py`, exported
internally for testing but not part of the public surface.

The transport stores `self._retry: RetryPolicy | None`, set from the
`MaxClient` constructor.

## Per-method `idempotent` flags

Each `MaxClient` method passes the right flag at the call site. The
calling method knows whether it is safe to replay; the transport does
not need to infer it.

| Method | `idempotent` |
|---|---|
| `send_message` | `False` |
| `edit_message` | `False` |
| `delete_message` | `False` |
| `get_messages` | `True` |
| `get_updates` | `True` |
| `get_chat` | `True` |
| `request_upload_url` | `True` |
| `upload_file` (presigned POST) | `False` |
| `upload_image` / `upload_video` / `upload_audio` / `upload_file_attachment` | inherit from the two underlying calls — `request_upload_url` is `True`, the actual `upload_file` is `False` |

## Testing strategy

New file `tests/test_retry.py` covers the policy itself. Existing
`tests/test_transport.py` (or equivalent) gains coverage for the loop
behavior. Selected `tests/test_client.py` cases pin per-method
`idempotent` flags.

### `RetryPolicy` unit tests

- Default values match the spec.
- `_backoff(policy, N)` with `jitter=False` yields the documented
  sequence; saturates at `backoff_max`.
- `_backoff(policy, N)` with `jitter=True` stays inside `[0.75 * base,
  1.25 * base]` over many samples.

### Transport-level tests (with `respx`, `asyncio.sleep` monkey-patched
to a no-op so suite stays fast)

- `policy=None`: zero retries on any failure — exact v0.1 behavior.
- `policy=RetryPolicy(max_attempts=3)`:
  - 5xx + `idempotent=True` → retries up to 3 attempts, succeeds on Nth.
  - 5xx + `idempotent=False` → no retry, raises immediately on first
    failure.
  - `MaxTransportError` → retries regardless of `idempotent`.
  - 429 with `Retry-After: 60` → policy sleeps ≥60s even when
    `backoff_max=30`.
  - 429 without `Retry-After` → falls back to computed backoff.
  - 4xx (400/401/404) → no retry, raises immediately.
  - All attempts fail → original exception type re-raised on the last
    attempt; no exception wrapping.

### Client-level tests

- `MaxClient.send_message` against a 503-then-200 mock with a policy
  set: raises `MaxServiceUnavailableError` on the first attempt (no
  retry — non-idempotent).
- `MaxClient.get_chat` against the same 503-then-200 mock with a policy
  set: succeeds on the second attempt.
- `MaxClient.upload_image` against `request_upload_url` failing 503
  twice then 200: succeeds (request_upload_url is idempotent).
- `MaxClient.upload_image` against the presigned POST failing 503: no
  retry (the upload step is non-idempotent).

### Coverage

Keep the global gate at 90%. `src/max_bot_api/retry.py` should hit
~100% line coverage.

## Module layout

- New: `src/max_bot_api/retry.py` — `RetryPolicy`, internal `_backoff`.
- Modified: `src/max_bot_api/transport.py` — new `idempotent` keyword on
  `request`, retry loop, `self._retry` storage.
- Modified: `src/max_bot_api/client.py` — accept `retry` in
  `__init__`, pass `idempotent=` at every call site (table above).
- Modified: `src/max_bot_api/__init__.py` — re-export `RetryPolicy`,
  add to `__all__`.
- New: `tests/test_retry.py` — policy and `_backoff` unit tests.
- Modified: `tests/test_transport.py` — loop behavior tests.
- Modified: `tests/test_client.py` — per-method idempotency tests.
