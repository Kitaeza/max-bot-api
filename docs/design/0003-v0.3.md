# python-max-bot v0.3 — Design

**Status:** approved
**Date:** 2026-05-04
**Author:** Kitaeza
**AI co-author:** Claude (Anthropic)
**Tracks:** Roadmap §v0.3+ in `docs/design/0001-v0.1.md`

## Goal

Bundle the three small endpoint groups from the v0.3+ roadmap into a
single release: webhook subscription endpoints, bot introspection
methods, and action indicators. Each is a thin wrapper over a single
Max API call; together they round out the "useful, predictable client"
promise without adding a new architectural layer.

## Non-Goals

- No webhook receiver server / dispatcher. We expose the three
  subscription endpoints (`subscribe`, `get_subscriptions`,
  `unsubscribe`) so users can register an HTTPS endpoint they own;
  building an HTTP server stays out of scope (deferred to v1.0+).
- No member-mutation endpoints (`POST /chats/{chatId}/members`,
  `DELETE /chats/{chatId}/members`, admin add/revoke). The roadmap
  names only the GET versions; flag mutations as a v0.4 candidate if a
  user asks.
- No auto-paginating helpers (`iter_chat_members`). Single-page
  cursor-based methods only, matching `get_updates`. Add later if
  asked.

## API surface

All methods land on `MaxClient`. All optional arguments are
keyword-only. All methods are async.

### Bot introspection

```python
async def get_me(self) -> BotInfo: ...

async def get_chat_members(
    self,
    chat_id: int,
    *,
    user_ids: list[int] | None = None,   # filter mode; ignores marker/count
    marker: int | None = None,
    count: int = 20,                      # 1..100, validated locally
) -> ChatMemberList: ...

async def get_chat_admins(
    self,
    chat_id: int,
    *,
    marker: int | None = None,
) -> ChatMemberList: ...

async def get_my_chat_membership(self, chat_id: int) -> ChatMember: ...
```

`get_my_chat_membership` wraps `GET /chats/{chatId}/members/me` —
returns the bot's own membership for that chat (admin status,
permissions, join time). Useful for "am I admin here?" preflight checks
before doing admin-only operations.

### Webhook subscriptions

```python
async def get_subscriptions(self) -> list[Subscription]: ...

async def subscribe(
    self,
    *,
    url: str,
    update_types: list[UpdateType] | None = None,
    secret: str | None = None,
) -> None: ...

async def unsubscribe(self, *, url: str) -> None: ...
```

The library does not run a webhook server. These methods only register
or unregister an HTTPS endpoint that the user operates themselves.

### Action indicators

```python
async def send_action(self, chat_id: int, action: ChatAction) -> None: ...
```

`ChatAction` is an enum of the five values the API documents
(`TYPING_ON`, `SENDING_PHOTO`, `SENDING_VIDEO`, `SENDING_AUDIO`,
`SENDING_FILE`).

### Validation rules (local, before HTTP)

- `subscribe(url=...)` — raises `ValueError` if `url` does not start
  with `https://`. The API requires HTTPS; surface earlier.
- `subscribe(secret=...)` — raises `ValueError` if not in `5..256`
  characters or contains characters outside `[A-Za-z0-9-]`. The API
  requires this exact charset; surface earlier with a clear message.
- `get_chat_members(count=...)` — raises `ValueError` if outside
  `1..100`. Matches the API's documented range.

### New file: `src/max_bot_api/models/subscriptions.py`

```python
class Subscription(BaseModel):
    """Active webhook subscription as returned by GET /subscriptions."""
    model_config = ConfigDict(extra="ignore")
    url: str
    time: int
    update_types: list[str] | None = None  # API uses strings; not enum-validated
```

`update_types` deliberately uses `list[str]` on the response model
rather than `list[UpdateType]` — the API may add new update types
server-side, and we don't want a forward-compatible enum mismatch to
crash a `get_subscriptions` call. On the request side (`subscribe(...)`)
we accept the strict `list[UpdateType]` because the user controls the
input.

### Internal: `_SimpleResponse`

Lives in `src/max_bot_api/models/_internal.py` (new file). Used only
internally to parse `{success, message}` envelopes returned by the
mutator endpoints (`POST /subscriptions`, `DELETE /subscriptions`,
`POST /chats/{chatId}/actions`).

```python
class _SimpleResponse(BaseModel):
    model_config = ConfigDict(extra="ignore")
    success: bool
    message: str | None = None
```

Not exported from the package root.

## Error handling

One new exception subclass:

```python
class MaxBadResponseError(MaxError):
    """The API returned 2xx with `{success: false}`.

    Synthesized from the response body — does not carry an
    httpx.Response, since the underlying HTTP call succeeded. Raised
    from subscribe / unsubscribe / send_action when the server reports
    a soft failure on an otherwise-successful HTTP response.
    """

    def __init__(self, message: str | None) -> None:
        super().__init__(message or "API reported success=false with no message")
        self.message = message
```

Inherits from `MaxError` (not `MaxAPIError`) — `MaxAPIError` requires a
`response: httpx.Response` field for transport-level errors, which we
don't have here since the HTTP call itself was 2xx. `MaxError` is the
right parent — users who want to catch "anything from this library"
catch `MaxError`; users who want HTTP-specific errors catch
`MaxAPIError`. This keeps both contracts clean.

Lives in `exceptions.py`. Re-exported from package root.

The mutator methods (`subscribe`, `unsubscribe`, `send_action`) parse
the response body into `_SimpleResponse` via the existing
`response_model` mechanism. If `success=false`, they raise
`MaxBadResponseError(resp.message)`. If the API returns proper 4xx/5xx
for these errors, the existing transport mapping catches it first and
`MaxBadResponseError` never fires — this is a safety net for the
documented payload shape. No change to `Transport` is required.

## Idempotency

Each method declares its idempotency at the call site, per the existing
v0.2 retry-policy contract.

| Method | `idempotent` |
|---|---|
| `get_me` | `True` |
| `get_chat_members`, `get_chat_admins`, `get_my_chat_membership` | `True` |
| `get_subscriptions` | `True` |
| `subscribe`, `unsubscribe` | `False` |
| `send_action` | `False` |

`subscribe` and `unsubscribe` are mutators (creating/removing config),
so safe-by-default applies even though same-URL replays are naturally
idempotent on the server.

`send_action` is theoretically harmless to replay, but we mark it
non-idempotent for consistency with all other writes — a typing
indicator that didn't reach the server is a no-op nobody will notice,
and the cost of *not* retrying is zero.

## Testing strategy

One test file per concern (matches the existing split). All tests
remain unit tests using `respx`; no live API hits in CI.

### New / extended test files

- **`tests/test_models_bot.py`** (new) — `BotInfo` and `BotCommand`
  round-trip; required vs optional field handling.
- **`tests/test_models_subscriptions.py`** (new) — `Subscription`
  round-trip; tolerates unknown `update_types` strings.
- **`tests/test_models_chats.py`** (extend) — `ChatMember` round-trip
  with and without admin-only fields; `ChatMemberList` shape;
  `ChatAdminPermission` and `ChatAction` enum coverage.
- **`tests/test_client_introspection.py`** (new) — `get_me`;
  `get_chat_members` happy / paginated (marker round-trip) / `user_ids`
  filter / `count` validation; `get_chat_admins`;
  `get_my_chat_membership`.
- **`tests/test_client_subscriptions.py`** (new) — `get_subscriptions`
  happy; `subscribe` happy + URL validation + secret validation +
  `success=false` → `MaxBadResponseError`; `unsubscribe` happy +
  `success=false` → `MaxBadResponseError`.
- **`tests/test_client_actions.py`** (new) — `send_action` happy for
  each enum value; `success=false` → `MaxBadResponseError`.
- **`tests/test_exceptions.py`** (extend) — `MaxBadResponseError <:
  MaxError` (and explicitly *not* `MaxAPIError`, since the HTTP call
  was 2xx and there is no response object to carry).

### Idempotency wiring

Add one extra case to `tests/test_client_retry.py`:

- `get_me` against a 503-then-200 mock with a policy set: succeeds on
  the second attempt (proves `idempotent=True` reaches the transport).
- `subscribe` against a 503 mock with a policy set: raises immediately
  (proves `idempotent=False`).

Coverage gate stays at 90%.

- Webhook receiver server / dispatcher framework (v1.0+ topic).
- Member-mutation endpoints (POST/DELETE members, POST/DELETE admins).
- `iter_chat_members` / `iter_chat_admins` async-iterator helpers.
- Live integration test workflow — separate brainstorm cycle after
  v0.3 ships, with a real bot token provided by the user.
